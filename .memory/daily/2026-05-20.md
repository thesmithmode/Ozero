### Session (KB query — ByeDPI IDLE root cause)

**Context:** Knowledge base query — причины IDLE/нет трафика в hev pipeline ByeDPI

**Key Exchanges:**
- Запрос: `ByeDPI трафик не идёт hev pipeline IDLE root cause`
- Синтез из 7+ KB статей: cascade root causes ранжированы по частоте

**Root Cause Summary:**
Первичные (самые частые):
1. `jniStartProxy=-1` от stale `server_fd` (unconditional `forceClose` отсутствует в stop/failure paths) → hev нет SOCKS upstream → IDLE
2. `setBlocking(true)` на TUN fd → замораживает epoll loop libhev → счётчик трафика stuck at 0
3. `excludeSelf=false` → трафик Ozero входит в TUN → hev роутит через ciadpi → обратно в TUN → бесконечный loop

Вторичные:
- Неправильный stop order (libhev раньше byedpi) → брошенные потоки держат mutex → deadlock при reconnect
- `g_proxy_running` guard wedged → следующий `jniStartProxy` возвращает -2 (GUARD_BUSY)
- `-Ku` в ByeDPI args → UDP-only whitelist, TCP/TLS desync не применяется

**Action Items (без изменений):**
- v0.1.8: revert setMtu + WARP sockets/ discovery — dirty файлы в репо, работа прервана компакцией

### Session (post-compact) — ByeDPI double-argv[0] regression

Зафиксировано в memory `byedpi-args-parsing.md` (2026-05-13). Регрессия введена `7cc2348f FIX: ByeDPI ciadpi argv[0]` — fix без проверки native-lib.c.

**Fixes Applied:**
- `TunBuilderHelper.kt` — убрал `setMtu(8500)` из buildTunBuilder
- `ByeDpiEngine.kt:252` — убрал `"ciadpi"` из prefix
- `ByeDpiUiArgsBuilder.kt:10` — убрал `"ciadpi"` из prefix
- `ByeDpiEngineTest.kt:258-268` — sentinel переписан: **запрещает** ciadpi/byedpi в Kotlin-argv (защита от регрессии)
- `ByeDpiUiArgsBuilderTest.kt:15-23` — expected обновлён без ciadpi
- `WarpHandshakeUapi.kt` — `findUapiSocket()` ищет в `sockets/` subdir + любой `.sock` через listFiles (canonical path amneziawg-go)

### Session (post-compact) — VPN slot release + ×2 quota архитектурный аудит

**Context:** Юзер: "VPN slot контракт Android" — можно обойти? + "×2 квота — ты уверен что server-side? мы должны генерить per-user аккаунт".

**Key Finding:**
- ×2 квота: per-user identity у Ozero уже корректная (Ed25519 keypair → walletAuth → отдельный URnetwork network с отдельным `byJwt`). Payout wallet = твой Solana адрес, привязывается через `walletVc.payoutWallet` отдельно (не identity). Архитектурно ×2 НЕ из-за shared account.
- Возможная причина: legacy migration (старый guest byJwt + walletAuth networkCreate подряд) → backend имеет 2 network на той же identity. Или backend накопил по networkId за всё время. Без логов точно не сказать.
- VPN slot: НЕ данность. `onRevoke()` ловит когда другой VPN перехватывает slot. Можно добавить отложенный `Process.killProcess` → процесс умирает, libgojni выгружается, slot моментально освобождается. Только в `onRevoke` (не stopVpn/onDestroy): revoke = юзер явно выбрал другой VPN, наша фоновая работа не нужна.

**Fixes Applied:**
- `OzeroVpnService.onRevoke()` — `Handler.postDelayed { Process.killProcess(myPid) }` через REVOKE_KILL_DELAY_MS=2500ms (даёт время shutdown+logs)
- `OzeroVpnServiceLifecycleTest` — sentinel: onRevoke содержит processKiller.kill + postDelayed + REVOKE_KILL_DELAY_MS ≥ 1500ms
- `RealUrnetworkSdkBridge.fetchSubscriptionBalance` — diagnostic GB-форматирование + ratio balance/start + WARN если startBalance > 50GB (DOUBLE_QUOTA_THRESHOLD_BYTES) с clientId

**Lessons Learned:**
- Не путать identity (Ed25519 pubkey) с payout wallet (Solana адрес). Ozero корректно разделяет.
- onRevoke ≠ stopVpn. Revoke значит external override → free process resources. Stop значит юзер вручную выключил VPN → процесс остаётся для UI/balance refresh.
- Каждое "это server-side, ничего нельзя" требует проверки 2-го уровня: показать архитектурную причину или предложить client-side investigation hook (diagnostic warn).
