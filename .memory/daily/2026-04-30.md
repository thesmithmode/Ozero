## Manual Notes (v0.0.1 release cycle 2026-04-30)

### byedpi args parser — критичные ловушки

byedpi (`hufrea/byedpi`) парсит argv через getopt_long. Несколько неочевидных деталей, которые мы выясняли через 5-субагентный ресерч + чтение C-кода `main.c`/`extend.c`/`native-lib.c`:

1. **`-Ku` = `--proto u` = whitelist протокола UDP-only**. Если в args стоит `-Ku`, то весь TCP/TLS трафик идёт **БЕЗ применения desync** — как обычный SOCKS5. Это означает что ТСПУ/DPI режет TLS handshake, а byedpi даже не пытается обходить. Симптом: tunnel up, byte stats растут (TCP SYN/ACK), но HTTPS виснет. Правильно: `-Kt,h` (HTTP+HTTPS) или `-Kt,h,u` (всё) или вообще без `-K` (default = всё).

2. **`-An` = auto-group с детектом `none` = fallthrough**. Просто открывает новую desync-group без триггера. Применяется ко всем connections в этой группе.

3. **`-A` группы расшифровываются через флаги: `t`=torst, `r`=redirect, `s`=ssl_err, `n`=none, `c`=conn**. Это **триггеры** для следующего fallback strategy. `-At,r,s` = группа применяется при timeout/redirect/ssl error.

4. **Multi-strategy chains** через повторяющиеся `-a1` boundaries: `-s1 -q1 -a1 -Y -Ar -a1 -s5 -o2 -At -f-1 -r1+s -a1 -As -s1 -o1+s -s-1 -a1` — это 5+ phase chain, где каждый `-a1` начинает новую strategy group, byedpi пробует их по порядку при failure.

5. **Default args ByeDPIAndroid в UI-режиме** ≈ `-Kt,h -d1` (disorder pos=1 для HTTP+HTTPS). Это известный working baseline для большинства ТСПУ-провайдеров.

6. **Working preset для российских провайдеров** (verified): `-s1 -q1 -a1 -Y -Ar -a1 -s5 -o2 -At -f-1 -r1+s -a1 -As -s1 -o1+s -s-1 -a1` — multi-phase chain с TLS-record-split + fake + disorder + DISOOB.

### argv[0] = program name в getopt → "ciadpi" prefix критичен

`getopt_long` трактует `argv[0]` как имя программы и парсит флаги начиная с `argv[1]`. Если передать `argv = ["-p", "1080", "-Ku", ...]`, то `-p` будет потерян (считается program name), а `1080` — free positional argument.

ByeByeDPI и ByeDPIAndroid prepend'ят `"ciadpi"` как `argv[0]` + `--ip 127.0.0.1` явно. Мы изначально передавали без program name — часть args терялась. Fix в `ByeDpiEngine.buildArgs`:
```kotlin
return (listOf("ciadpi", "--ip", "127.0.0.1", "-p", config.socksPort.toString()) + extra).toTypedArray()
```

### MTU dual-layer: Builder vs hev YAML — разные слои

Это ключевая ловушка для libhev-socks5-tunnel + Android VpnService:

- `VpnService.Builder.setMtu(N)` = **link MTU cap для TUN-интерфейса**. Android-приложения шлют пакеты до N байт в TUN. На cellular path MTU обычно ~1500.
- `socks5-tunnel YAML mtu: N` = **lwIP internal buffer cap**. Это размер сегментов которые lwIP формирует при reassembly из TUN-пакетов. Никак не выходит наружу.

ByeByeDPI/ByeDPIAndroid:
- `Builder.setMtu()` **НЕ вызывают** (Android берёт link MTU sane = 1500)
- YAML `mtu: 8500` (для lwIP performance, не для path)

Мы изначально выставляли `setMtu(8500)` И `mtu: 8500` в YAML — приложения шлют 8500-байтные пакеты в TUN, lwIP пере-генерирует TCP сегменты, kernel TCP пытается отправить их через cellular interface где path MTU ~1500 → fragments → drop → retransmit storm.

Симптом: загрузка идёт **рывками** (bursts + drops). Δrx 5MB → 1KB → 6MB → 100KB pattern.

**Fix**: убрать `setMtu()` из Builder вообще, оставить YAML `mtu: 8500`. Паритет с ByeByeDPI.

### IPv6 unconditional → byedpi не поддерживает → silent drop

Если в TUN добавлены IPv6 routes (`addAddress(fd00::1/128) + addRoute("::", 0)`), Android dual-stack приложения шлют v6 пакеты в TUN. libhev forwarding'ит их через SOCKS5 к byedpi, но byedpi не делает v6 outbound — silent drop. Браузеры таймаутят на v6 → fallback на v4 → tail latency на каждый запрос.

ByeByeDPI добавляет v6 только если `ipv6_enable=true` (default false). Мы добавляли безусловно.

**Fix**: убрать v6 routes из всех `SplitTunnelMode` (ALL/BYPASS_LAN/ALLOWLIST/BLOCKLIST). Условный IPv6 toggle в Settings — отдельная фича на v0.0.2.

### setMetered(false) on API≥Q — против OEM throttle

Nubia/RedMagic ROM агрессивно урезают throughput для metered VPN. ByeByeDPI вызывает `Builder.setMetered(false)` (Q+). Без этого VPN considered metered → ОС background throttle → нестабильная скорость.

### EWMA для bps в UI — сглаживание TCP burst jitter

`TProxyGetStats()` cheap (LongArray return из RegisterNatives), 1Hz sampling безопасен. Raw delta между двумя samples даёт jitter (TCP burst pattern). Сглаживание EWMA α=0.4:
```
smoothed = α * rawDelta + (1-α) * prev
```
Один Double поле, дешевле moving average buffer. Считать в `TunnelController.updateStats` (не в Compose recomposition).

### Stats native counters — auto-reset на tunnel restart

`hev_socks5_tunnel_get_stats(tx_packets, tx_bytes, rx_packets, rx_bytes)` возвращает **кумулятивные счётчики за время жизни tunnel**. Native сам обнуляет на каждый `TProxyStartService`. Это **значит session bytes = raw native** — никакого virtual reset в Kotlin не нужно. Простота и бесплатно.

### v0.0.1 success findings — суммарно что починило DPI bypass

Чейн фиксов (4 retag цикла):
1. **`setBlocking(true)` → удалить** (Phase A1, 2026-04-30 утром): libhev event-loop делает epoll-async I/O, blocking mode замораживал → traffic 0.
2. **Stop order инверсия** (Phase A2): byedpi → libhev (а не наоборот). libhev event-loop ждёт outstanding socks5 connections к byedpi, abandoned native thread держит singleton mutex → reconnect deadlock.
3. **DNS list 100.64.0.1 (CGNAT) → public** (Phase A3): 100.64.0.0/10 reserved range, никто не отвечает на устройстве.
4. **fd.dup() удалить** (Phase A4): передавать raw `tunPfd.fd` к native, паритет с ByeByeDPI/ByeDPIAndroid.
5. **Default args replace `-Ku -a1 -An -o1 -At,r,s -d1` → working preset** (Phase D1): `-Ku` блокировал desync для TCP.
6. **`setMtu(8500)` удалить** (retag-5): fragmentation на cellular.
7. **IPv6 routes удалить** (retag-5): byedpi silent drop v6.
8. **`setMetered(false)`** (retag-5): Nubia throttle.
9. **argv[0]="ciadpi" + `--ip` явный** (retag-5): getopt parsing.

Ключевой root-cause для каждого симптома: traffic-0 = `setBlocking`. Reconnect deadlock = stop order. Bursts = MTU + IPv6 + setMetered combo.
