### Session (12:06)

**Context:** ByeDpi engine reliability fix — stale server_fd causing cascade failures + Strategy UI cleanup

**Key Exchanges:**
- Юзер отверг поверхностный advisor-анализ, потребовал глубже вникнуть самостоятельно
- Найден root cause: `jniStopProxy` делает только `shutdown()` без `close()`/reset; failure path в `ByeDpiEngine.start()` пропускал `forceClose()` когда `proxyJob.isActive==false` → stale `server_fd` → cascade -1 returns
- Доказательство из прод-лога: 10 fails подряд, единственный `jniForceClose` (строка 15254) привёл к successful connect через 4 мин
- 0% fitness в EvolutionEngine = тот же root cause (600 start/stop циклов накапливают stale fd)

**Decisions Made:**
- Fix Kotlin-only без форка submodule: убрать conditional `if (proxyJob.isActive)` — всегда `join + forceClose` в start() failure path и stop()
- `forceClose` noop при `server_fd < 0` — safe для дубль-вызовов
- Инвертирован существующий тест `startFailureNoStopProxyWhenProxyReturnedErrorImmediately` (ранее доказывал баг как "expected behavior")
- Strategy UI: убран `DomainListsHeader` с главного экрана, перенесён в TopAppBar action icon, добавлен progress bar

**Lessons Learned:**
- `ByeDpiEngine.stop()` и `start()` failure path **всегда** должны вызывать `proxy.forceClose()` после `job.join()`, не conditional на `isActive`. Upstream byedpi main() хранит server_fd глобально — без close stale fd блокирует все следующие starts
- Тест что верифицирует "forceClose NOT called" может быть sentinel бага, а не корректного поведения — проверять семантику assertion
- Юзер требует глубокого самостоятельного исследования, не advisor-гадание на неполных данных
