### Session (11:01)

**Context:** Диагностика deadlock в GoRuntimeGuard при быстрой смене движков (WARP/URnetwork)

**Key Exchanges:**
- Q: Что ломается при быстрой смене движков?
- A: GoRuntimeGuard deadlock: teardown-корутина отменяется → `release(owner)` не вызывается → следующий `acquire()` блокируется навсегда. 7 `startVpn` за 30 сек = 2+ leaked owners = systematic deadlock.

**Decisions Made:**
- Fix: удалить GoRuntimeGuard полностью. Guard — symptom-patch поверх уже решённой проблемы: eager loading в `OzeroApp.onCreate` делает concurrent init/teardown невозможным. Guard добавляет deadlock-риск, не предотвращая ничего реального.

**Lessons Learned:**
- Eager loading (оба Go-рантайма resident навсегда) устраняет сценарий, от которого защищал GoRuntimeGuard. Guard стал ложным слоем защиты с новым failure mode хуже исходного.
- Корутина teardown при отмене (`startVpn` вызывается снова) не завершает `release(owner)` → sticky owner = deadlock для любого следующего движка.
- Симптом "URnetwork сломан после переключения" + "WARP не стартует" = одна корневая причина (leaked guard owner), не два разных бага.

**Action Items:**
- Удалить `GoRuntimeGuard` из codebase
- Проверить все пути teardown на наличие парного `release(owner)` (или убедиться что guard удалён везде)
- Добавить sentinel-тест: быстрая смена движков (WARP→URnetwork→WARP) не должна deadlock-ить
