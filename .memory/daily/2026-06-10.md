## Memory Maintenance

### Session (13:08)

**Context:** Пользователь просил аналитически объяснить, почему релиз `1.0.13` имел зеленый CI, а текущая ветка стала постоянно красной из-за покрытия тестами.

**Key Exchanges:**
- Пользователь явно ограничил режим: только аналитика, ничего не менять.
- Было установлено различие между `v1.0.13` и текущим `dev`: в `v1.0.13` CI запускал тесты, но не использовал coverage verification как жесткий gate.
- Текущий CI вызывает `jacocoTestCoverageVerification` для нескольких job’ов и модулей: `buildSrc`, `app`, `common-vpn`, `engine-warp`, группы engine/singbox/core-модулей.

**Decisions Made:**
- Ответ дан через сравнение механизмов CI, а не через предположение о “плохих тестах”: причина в изменении критерия прохождения CI.
- Зафиксировано, что зеленый `1.0.13` и красный текущий CI не противоречат друг другу, потому что проверяют разные условия.

**Lessons Learned:**
- В текущем проекте coverage gate задан в `buildSrc/src/main/kotlin/ozero.jacoco.gradle.kts` с порогом `0.95` для line и branch coverage.
- Добавление `jacocoTestCoverageVerification` в CI превращает недостаточное покрытие любого проверяемого модуля в падение job’а.
- Исторически зеленый CI может означать только “тесты прошли”, если coverage verification тогда не был частью gate.

**Action Items:**
- При необходимости следующим шагом разложить историю по конкретным коммитам, которые ввели coverage gate и расширили список проверяемых модулей.
