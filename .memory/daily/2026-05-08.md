### Session (14:00) — Crash root + fix

**Корень SIGABRT URnetwork→WARP (по ozero.log 12732 строк):**
- В обоих tombstone (PID 32444, 7659) бэктрейс libam-go.so → `runtime.abort` + `runtime.gcWriteBarrier*` в DefaultDispatcher coroutine. Это Go-runtime fatal (corruption в heap/GC), не SIGSEGV nullptr.
- Лог за 8 минут показал 13 entry awgTurnOn / 11 exit awgTurnOff. 2 без exit = 2 краша.
- `RealWarpSdkBridge.attachTun` перезаписывал `tunnelHandle = handle` без turnOff предыдущего → старый handle висел в Go runtime → накопление → SIGABRT после ~5-10 циклов.
- Краш ВСЕГДА после повторного awgTurnOn в одном pid без парного awgTurnOff. Никогда после первого старта.

**Фикс (commit f7cdcdc на dev):**
- В начале `attachTun` если `tunnelHandle != INVALID_HANDLE` — закрыть стар. handle через runCatching turnOff, сбросить поле, потом продолжить. Idempotency-guard на уровне bridge.
- Sentinel-тесты: двойной attach без detach (turnOff=1), stale turnOff бросает (новый turnOn ОК), ×10 цикл (turnOff = turnOn − 1).

**Что НЕ корень (отвергнуто):**
- `project_warp_root_cause.md` (lossy parse + watchdog) — не применимо, raw INI passthrough печатает 22 ключа включая I1, parse не теряет.
- `project_engine_switch_crash.md` (stopSelf race) — не применимо, performShutdown begin/end в правильном порядке, нет ForegroundServiceDidNotStart.
- loadOnce thread=main инвариант не нарушен.

**Прочие подозрительные паттерны (не в фокусе сейчас):**
- HealthMonitor degraded (3 fail) → нет recovery action / engine restart. URnetwork висел 3 мин в degraded.
- URnetwork SOCKS5 на port=10810 не отвечает 4 мин при `Connected` — фиктивный success в EngineUrnetwork.start.
- `chainOrchestrator.stop completed` иногда дублируется (двойной stop без интервала). Симптом параллельного stopVpn из ACTION_STOP + onDestroy/restart.
- `tun interface discovery failed` 18 раз WARN — fallback работает (uid-stats), но шум на каждом старте.
- Tombstone-симы раздувают лог в 30x (binary protobuf печатается ASCII-извлечением). Парсить .pb отдельно.

**Lessons Learned:**
- Native runtime leak не виден ни в одном Java-стактрейсе. Counts entry/exit JNI вызовов в логе — единственный способ обнаружить парность.
- Go runtime SIGABRT с `runtime.gcWriteBarrier*` в backtrace ≈ corrupted heap, чаще всего от reentry без cleanup.
- Defensive idempotency-guard на bridge layer лучше любых "правильных" проверок выше — фикс работает независимо от того кто и почему вызвал attach дважды.
