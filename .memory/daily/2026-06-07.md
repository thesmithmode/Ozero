- Выяснилось, что локальные падения тестов на Windows в основном связаны не с логикой, а с блокировками `build/` и проблемами Gradle/daemon, то есть это ограничение окружения.
- На вопрос, стоит ли вообще чинить локальный прогон, был сделан вывод, что для поиска кодовых проблем надежнее опираться на GitHub Actions, а локально оставить только `ktlint/detekt` и точечные тесты.

**Decisions Made:**
- Перейти от локального full-suite подхода к модульным прогонам или CI-логам, потому что локальная среда на Windows дает ложные фейлы из-за file-lock’ов.

**Key Exchanges:**
- Уточнили, что красными были не только `app`, но весь matrix: `buildSrc`, `common-vpn`, `core + common`, `engine-urnetwork + engine-byedpi`, `engine-warp`, `singbox + extra modules`.

- Выяснилось, что часть падений была не в проде, а в тестах и контрактах: stale expectations, type inference, DataStore/state race, tolerant parser, parcel roundtrip.
- Согласовали, что сначала чинятся source-level failures, а coverage gates разруливаются только после возвращения unit tests в green.

**Decisions Made:**
- Основным источником истины взять `push`-run последнего SHA, а PR-run использовать для сверки.
- Сначала закрывать compile/unit failures, потому что coverage нельзя честно добивать на фоне красных тестов.
- Для FPTN store убрать stale initial state и синхронизировать in-memory snapshot с flow, чтобы UI и engine не видели старую конфигурацию.
- Для URnetwork store убрать сериализацию optional values через `JSONObject.NULL`, чтобы не ловить NPE и не оставлять мусор в cache.
- Для `common-vpn` обновить хрупкий source-based тест под уже введенный `hasBlockingTunForKillswitch()`.
- Для WARP перейти на явный parcel contract с marker-ами вместо зависимости от нестабильной Parcelable-сериализации.

**Lessons Learned:**
- `StateFlow` с initial default может давать stale read в CI и в runtime, если update не синхронизирует локальный snapshot.
- `JSONObject.NULL` в Kotlin/Java bridge может ломать non-null paths и лучше обходиться явными nullable/marker схемами.
- Source-assert тесты на строки кода быстро устаревают после рефакторинга и требуют обновления под фактический инвариант.
- В Android `Parcel`-контрактах лучше не опираться на deprecated/неявные сериализационные пути, особенно для nullable `ParcelFileDescriptor`.
- Для CI-расследования сначала нужно снять сигнатуры ошибок по всем jobs, иначе легко зациклиться на одном симптоме.
