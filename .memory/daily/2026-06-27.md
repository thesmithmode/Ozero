### Session (18:35)

**Context:** Доработка фиксов singbox/Room и WARP после красного CI и ревью по process-death handling.

**Key Exchanges:**
- Пользователь указал, что WARP P1 не закрыт: `onServiceDisconnected` может прийти без `onBindingDied`, а старый код не вызывал failure path через `RuntimeFailureRouter.handleEngineFailure`.
- Пользователь также указал на падающие CI checks `singbox + extra modules` и `singbox-room`.
- Было подтверждено, что старый WARP sentinel ошибочно закреплял поведение “не вызывать `onProcessDied`” в `onServiceDisconnected`.

**Decisions Made:**
- `onServiceDisconnected` для WARP теперь считается сигналом смерти service process и обязан вызывать `onProcessDied`, чтобы активный VPN сразу уходил в engine failure/killswitch path.
- Тест `singbox-room` заменён с runtime reflection проверки `ColumnInfo` на source/schema sentinel, потому что `ColumnInfo` не runtime-visible и reflection даёт ложный сигнал.
- Старые misleading тексты тестов/логов про “recovery will rebind” обновлены, чтобы не закреплять неверную модель поведения.

**Lessons Learned:**
- Room `ColumnInfo` нельзя надёжно проверять через runtime reflection из-за retention; для таких контрактов нужен source/schema-based sentinel.
- `onServiceDisconnected` в Android binder lifecycle может быть единственным callback при убийстве service process под memory pressure; нельзя полагаться только на `onBindingDied`.
- Sentinel tests могут закреплять старый баг как “ожидаемое поведение”, их нужно пересматривать при ревью process-death сценариев.

**Action Items:**
- Следить, чтобы будущие WARP lifecycle изменения не обходили failure router при смерти subprocess/service process.
- При похожих Room schema изменениях проверять entity annotation target и фактическую Room schema/migration contract, а не только Kotlin property-level annotation.
