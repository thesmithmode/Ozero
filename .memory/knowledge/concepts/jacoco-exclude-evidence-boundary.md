# JaCoCo exclude evidence boundary

## Key Points
- Coverage exclusions must be justified by concrete untestability, not by broad class-name suffixes.
- Masks such as `*Engine*`, `*Config*`, `*Parser*`, `*Coordinator*`, `*Client*`, `*Service*`, `*Runtime*`, `*Bridge*`, and `*Proxy*` can hide testable production code.
- `jacocoTestCoverageVerification` must stay in CI; removing it creates false green builds.
- Generated bytecode, Android glue, manifest/resource wrappers, JNI loader shells, and external SDK bridges can be excluded only with narrow evidence.

## Details

The 2026-06-01 CI session confirmed that Ozero's 95% coverage requirement is a hard gate. A green CI after removing `jacocoTestCoverageVerification` does not prove the coverage contract. Likewise, sweeping excludes by class name can remove exactly the parser, engine, config, provider, and coordinator code that should be protected by regression tests.

The durable boundary is evidence-based: exclude only code that is objectively outside meaningful JVM/unit coverage, such as generated wrappers, Android lifecycle/resource glue, native binary loading shells, subprocess adapters, or external SDK/SSH/UAPI bridges. Testable business logic remains in the measured surface and should be covered by targeted contract tests.
