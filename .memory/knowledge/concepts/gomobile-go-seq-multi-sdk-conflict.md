# gomobile go.Seq Conflict Between Multiple SDKs

When two gomobile-generated SDKs (e.g., URnetwork SDK and libbox/sing-box) are loaded in the same Android process, their `go.*` Java classes (`go.Seq`, `go.Seq.Proxy`, etc.) conflict. The JVM loads `go.Seq` from whichever AAR was loaded first; the second `.so` calls `GetMethodID` expecting its own signature layout but finds the other SDK's version → `JniAbort` → SIGSEGV.

## Key Points

- gomobile-generated `.so` files each expect a specific version of `go.*` Java classes that match the Go runtime they were compiled with
- Loading two gomobile SDKs' `go.*` classes in the same DEX → one `.so` gets incompatible JNI method IDs → crash
- Tombstone shows: `Libbox.<clinit>` → `SetupOptions.<clinit>` → `SingboxRuntime.setup` → SIGSEGV
- Fix: strip `go.*` from one AAR (in CI), provide matching go-stubs.jar as `implementation` for the other
- `compileOnly` go-stubs does NOT go into runtime DEX — must be `implementation` for the classes to be available at runtime

## Details

### Crash Mechanism

In Ozero, `URnetworkSdk.aar` contains `go/*.class` files generated by gomobile bind. `libbox.aar` (sing-box) also contains `go/*.class` from its own gomobile build. When both AARs are merged into the app's DEX, the JVM has one set of `go.*` classes — whichever was processed first.

At runtime, `libbox.so` native code calls into JVM via `JNI_OnLoad` → registers methods against `go.Seq.Proxy`. If the `go.Seq.Proxy` in DEX came from URnetwork's build, field offsets and method IDs are different from what `libbox.so` expects → `GetMethodID` fails → `JniAbort` → SIGSEGV.

The tombstone process is in `:engine_singbox`, with the abort occurring within 72ms of `startWithConfig` being called.

### Fix Architecture

1. **Strip `go.*` from `URnetworkSdk.aar`** in CI workflows (`ci.yml`, `release.yml`) — keeps only libbox-compatible `go.*` in DEX
2. **Provide matching `go-stubs.jar`** extracted from the libbox AAR build (the `go/*.class` files compatible with `libbox.so`)
3. **`implementation`** (not `compileOnly`) for go-stubs in `singbox-core` — classes must be present in runtime DEX, not just at compile time
4. Each consuming module that extends `go.Seq.Proxy` must also have `compileOnly(go-stubs)` — see [[concepts/compileonly-nontransitive-android-library]]

```bash
# CI step: strip go.* from URnetwork AAR
cd engine-urnetwork/libs
jar xf URnetworkSdk.aar
jar uf URnetworkSdk.aar --exclude 'go/*.class'
```

### Process Isolation Prevents the Problem in VPN Process

Since URnetwork and singbox run in separate processes (`:app` vs `:engine_singbox` vs `:engine_urnetwork`), the conflict only manifests if both SDKs end up in the same process's DEX. The fix ensures the main app process DEX has libbox-compatible `go.*` and URnetwork's classes do not override them.
