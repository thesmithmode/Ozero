# Codex PR review threads should drive verified fixes

## Key Points
- Codex PR review comments are hypotheses until each thread is checked against current code and runtime contracts.
- Review handling should be thread-aware; flat comment lists can mix outdated comments with unresolved findings.
- Real P2 findings should receive minimal production fixes plus regression coverage before being pushed to `dev`.
- PR or `main` actions remain blocked unless the user explicitly asks for them.
- `.memory` daily updates generated by hooks should be included in the nearest related work commit.

## Details

The PR #80 review loop on 2026-05-31 showed a practical workflow for external AI review. The assistant first inspected unresolved Codex threads, verified each finding against code, and only then implemented fixes. The first wave covered ByeDPI detached short-option values, coalesced restart state loss, sing-box latency refresh while connected, and EvolutionEngine timeout sizing. A later wave covered quoted SNI normalization, ByeDPI connected-session IP lookup, and sing-box routed readiness.

The loop also clarified process boundaries. The work was pushed to `dev`, CI and PR CI were monitored to success, and unresolved review threads were rechecked. No `dev -> main` PR or merge was created because project rules require an explicit command before touching `main`.
