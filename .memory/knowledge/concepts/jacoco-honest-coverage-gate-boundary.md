# JaCoCo honest coverage gate boundary

## Summary
Ozero's 95% coverage gate is a hard release signal. It must not be bypassed by removing `jacocoTestCoverageVerification` or by broad class-name excludes that hide testable production code.

## Key Points
- Coverage 95% is mandatory, not a ratchet target or advisory report.
- Removing `jacocoTestCoverageVerification` from CI creates false green CI.
- Broad excludes like `*Engine*`, `*Config*`, `*Parser*`, `*Coordinator*`, and `*Client*` can hide exactly the production logic that needs tests.
- Exclusions require concrete non-unit-testable reasons, such as generated bytecode, Android manifest/resource glue, JNI/native binary shell, or external bridge code.
- Parser/config/model/runtime logic should stay in coverage and receive regression or contract tests.

## Details
During the June 1 CI recovery, one attempted direction removed coverage verification from CI and added wide suffix masks to JaCoCo exclusions. The user rejected that as violating the project contract. The correction restored `jacocoTestCoverageVerification` and removed broad masks that could hide engine, parser, config, provider, coordinator, and client code.

The accepted boundary is not "low value by class name"; it is "objectively not unit-testable by role and evidence." Compose/UI shells, generated classes, precompiled Gradle wrappers, Hilt/generated artifacts, JNI loaders, subprocess/native shells, and external SDK bridges can be excluded only when the specific bytecode is a measurement artifact or boundary wrapper. Business logic and parse/validation behavior must remain measurable and protected by tests.

This concept extends [[concepts/android-jacoco-executiondata-false-green]] by distinguishing false green execution data from false green gate removal. It also links to [[concepts/ci-coverage-gate-artifact-trust-contract]] because a trustworthy gate must consume real execution artifacts and keep meaningful production code in scope.
